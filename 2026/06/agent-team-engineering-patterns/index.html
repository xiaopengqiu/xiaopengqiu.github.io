Agent Team 工程范式：从 while 循环到自治协作

Anthropic 的《Building a C compiler with a team of parallel Claudes》真正值得学习的地方，不是”AI 写出了 C 编译器”这个结果，而是背后一套可以复用的 Agent Team 工程范式：用极简循环让 agent 长时间运行，用共享 Git 仓库和文件锁完成协作，用高质量测试和结构化日志提供反馈，再把不可并行的复杂任务拆成可并行的小任务。

原文地址：Building a C compiler with a team of parallel Claudes

一、核心思想：不是聊天，而是持续执行

传统 LLM 使用方式是”人问一句，模型答一句”。Claude Code 这类工具把模型推进到 pair programming：它能读代码、改文件、跑测试，但默认仍然会在很多节点停下来等人确认。

这个脚本的关键不在 bash 技巧，而在它改变了 agent 的工作形态：

  1. while true 让每次会话结束后立刻启动下一次，模型没有”等用户继续”的默认退路。
  2. prompt 放在 AGENT_PROMPT.md 中，意味着 agent 的行为规范可以像代码一样版本化。
  3. 日志按 commit hash 命名，方便回溯每一轮 agent 从哪个状态开始、做了什么、失败在哪里。
  4. --dangerously-skip-permissions 取消交互确认，使无人值守成为可能，但也要求必须在容器中运行。

这里的范式可以概括为：让 agent 每轮只负责局部推进，但系统层面通过循环保证整体持续前进。

这和人类团队很像。一个工程师每天只解决有限的问题，但如果任务板、代码仓库、CI 和文档持续存在，整个项目就能长期演进。Agent Team 要做的，就是给模型搭出类似的工程环境。

二、协作拓扑：共享 Git，而不是中心调度器

很多 multi-agent 框架会引入一个 orchestrator：中心 agent 拆任务、派任务、收结果。作者选择了更朴素的方案：没有中心调度器，所有 agent 平等自治。

整体结构大致是：

每个 agent 都有独立容器、独立文件系统、独立进程空间。它们通过 Git 同步代码，通过仓库里的文档理解状态，通过锁文件认领任务。完成后再 pull、解决冲突、push 回 upstream。

这个设计有几个好处：

  • 简单：不需要 Redis、数据库、消息队列或复杂调度服务。
  • 可审计：每次修改都在 Git 历史里，agent 的协作痕迹天然可追踪。
  • 容错：某个 agent 容器挂掉，不会破坏其他 agent 的运行环境。
  • 自治：每个 agent 可以根据当前仓库状态自己决定下一步，而不是等待中心指令。

这套结构的本质，是把 Git 当作 multi-agent 系统的共享内存和一致性边界。

三、锁机制：用文件系统做任务认领

并行 agent 最大的问题是重复劳动：16 个 agent 如果都看到同一个失败测试，很可能同时去修同一个 bug。作者的解决方案非常直接：在仓库中放一个 current_tasks/ 目录，每个 agent 开始任务前写入一个文本文件作为锁。

锁文件可以包含：

  • 任务目标；
  • 当前假设；
  • 已尝试但失败的方法；
  • 相关测试命令；
  • 预计修改范围。

如果两个 agent 同时抢同一个任务，Git push 会暴露冲突或拒绝其中一方。失败的 agent 重新拉取最新状态后，就能看到任务已经被认领，转而选择其他任务。

这不是严格意义上的分布式锁服务，但在这个场景里足够有效。原因是 agent 的任务粒度通常以分钟到小时计，不需要毫秒级锁竞争；Git 的原子 push 已经提供了足够好的冲突检测。

更重要的是，锁文件本身也是上下文。下一个 agent 不只知道”这个任务被占了”，还知道”谁在做、为什么做、已经试过什么”。

四、自治决策：让每个 agent 自己找下一步

这篇文章里最反直觉的一点，是作者没有设计一个全局任务分配系统。每个 agent 启动后，需要自己阅读：

  • README；
  • progress 文档；
  • 当前失败测试；
  • current_tasks/ 锁文件；
  • 最近提交；
  • 测试日志。

然后它自己判断：”现在最值得推进的事情是什么？”

这种自治决策的前提，是仓库必须被设计成 agent 友好的环境。也就是说，人类不能只写给人看的文档，还要写给下一轮模型看的状态文件。

这类文档的价值，是把 agent 的短期记忆外化为项目状态。单个 Claude 会话会结束，上下文窗口会清空，但仓库里的文档会留下来。

五、反馈系统：测试是 agent 的方向盘

自治 agent 最大的风险不是”它不会工作”，而是”它很努力地解决了错误的问题”。

因此，Agent Team 的核心不是循环脚本，而是测试与反馈系统。作者明确提到，他的大部分精力花在设计 agent 周围的环境：测试、日志、构建脚本、验证器和 CI。

对 agent 来说，测试不只是质量保障，更是任务定义。你给它什么测试，它就会朝什么方向优化。如果测试模糊、误报、漏报，agent 会非常认真地把错误目标做到极致。

一个适合 agent 的测试 harness，需要满足几条原则：

1. 输出短，但信息密度高

不要让测试命令在终端里刷几千行无意义输出。长输出会污染上下文，模型很容易在噪声中丢失重点。

更好的做法是：

  • 控制台只打印摘要；
  • 详细日志写入文件；
  • 错误行统一包含 ERROR；
  • 失败原因和文件路径放在同一行，方便 grep ERROR；
  • 预先计算通过率、失败分布、回归数量。

2. 提供 fast mode

Claude 没有稳定的时间感。如果每次都跑完整测试，它可能花几个小时等待，而没有意识到这对当前迭代不划算。

所以测试系统需要提供 --fast 模式：

  • 只跑 1% 或 10% 的样本；
  • 对同一个 agent 保持确定性，便于比较改动前后；
  • 不同 agent 的样本不同，整体并行覆盖更多空间；
  • 快速发现明显回归，再决定是否跑完整测试。

这相当于给 agent 一套便宜的局部反馈，让它能高频试错。

3. CI 必须阻止回归

项目后期最常见的问题，是 agent 修一个失败测试时破坏已经通过的功能。此时单靠 prompt 提醒”不要破坏现有功能”是不够的，必须让 CI 成为硬约束。

有效的规则应该是：

  • 新提交不能降低核心测试通过率；
  • 已修复的回归测试必须固定下来；
  • 新功能必须带最小复现用例；
  • 对架构级修改要求跑更大范围测试。

Agent Team 不是不需要工程纪律，而是比人类团队更依赖工程纪律。

六、任务并行化：用 oracle 切开大问题

并行 agent 对”大量独立失败测试”非常有效。比如还有 300 个测试失败，每个 agent 认领一个失败用例，修完后提交，这就是天然并行。

但有些任务不是这样。编译 Linux 内核就是一个巨型问题：内核启动失败时，所有 agent 看到的是同一个崩溃现象。它们会同时追同一个 bug，互相覆盖修改，并行效率很低。

作者的关键做法，是引入 GCC 作为 oracle，把巨型问题切成小问题：

  1. 大部分内核文件先用 GCC 编译，认为这些对象文件是可信的。
  2. 只挑一小部分文件用 Claude 写的 C 编译器编译。
  3. 如果内核能启动，说明这批文件大概率没问题。
  4. 如果内核失败，就继续替换、二分、缩小范围。
  5. 最终把问题定位到具体文件，甚至具体文件组合。

这是一种非常重要的模式：当任务本身不可并行时，不要强行增加 agent，而是先设计一个能把任务切碎的验证器。

在其他领域也可以复用：

  • Web 迁移：用旧系统作为 oracle，对比新旧页面 HTML、截图、接口响应。
  • 数据处理：用老 pipeline 作为 oracle，对比抽样数据的结果差异。
  • 编译器/解释器：用成熟实现作为 oracle，做 differential testing。
  • API 重构：同时调用新旧接口，对比状态码、字段、边界行为。

好的 oracle 能把”系统整体不工作”变成”这 37 个样本里有 4 个不一致”，而后者才适合 agent 并行处理。

七、角色化：不是所有 agent 都去修 bug

当项目变大后，让所有 agent 都做同一种事情会浪费能力。作者后来引入了角色化 agent：

  • 去重 agent：查找重复实现，合并相似逻辑。
  • 性能 agent：优化编译器自身速度。
  • 代码生成 agent：提升输出目标代码质量。
  • Rust 评审 agent：从 Rust 工程设计角度批评结构。
  • 文档 agent：维护 README、进度、已知问题和使用说明。

角色化不是复杂的组织架构，而是不同 prompt 的结果。它的核心价值是让 agent 的注意力变窄：一个 agent 不需要同时关心所有问题，只要在一个维度上持续扫描项目。

这对实际工程很有启发。比如一个应用项目可以并行跑：

  • feature agent：按 issue 实现功能；
  • test agent：补充失败用例和回归测试；
  • refactor agent：降低重复和复杂度；
  • security agent：检查鉴权、输入校验、敏感信息；
  • docs agent：同步接口文档和运维手册。

它们不需要一个精密的会议机制，只要共享仓库、共享 CI、共享任务锁和清晰的状态文档。

八、Prompt 不是命令，而是操作系统规则

在普通对话里，prompt 是一次性指令。在 Agent Team 里，AGENT_PROMPT.md 更像一个小型操作系统规则，定义 agent 应该如何在环境中生存。

最小闭环：

  1. 每个 agent 从同一个 upstream clone。
  2. 启动后读取 AGENT_PROMPT.md 和 progress.md。
  3. 在 current_tasks/ 中创建锁文件。
  4. 执行小步修改。
  5. 跑 scripts/test_fast.sh。
  6. 通过后跑相关完整测试。
  7. 更新文档和日志。
  8. commit、pull、解决冲突、push。
  9. 外层 while true 启动下一轮。

这个版本不优雅，但可理解、可调试、可逐步增强。

十、边界：测试通过不等于可以上线

Agent Team 会制造一种危险幻觉：既然测试通过，代码也能跑，那是不是可以直接部署？

不能。

自治 agent 的产出尤其需要人类审查，原因有三点：

  1. 测试只覆盖你想到的行为，不能证明需求完整。
  2. agent 可能用脆弱、重复、难维护的方式让测试通过。
  3. 安全、性能、可观测性、运维成本往往不在单个测试里体现。

所以更合理的定位是：Agent Team 适合把大量可验证工作快速推进到”候选完成”状态，然后由人类做架构、安全和产品层面的验收。

它不是替代工程纪律，而是放大工程纪律。测试、CI、日志、隔离、任务拆分、代码评审越扎实，agent team 的效果越好；这些基础越薄弱，agent 只是更快地制造混乱。

总结

这篇 Anthropic 文章给出的不是一个复杂框架，而是一组朴素但强大的工程原则：

  • 用 while true 把 agent 从一次性助手变成持续执行者。
  • 用容器隔离危险能力。
  • 用共享 Git 仓库作为事实源。
  • 用 current_tasks/ 文件锁避免重复劳动。
  • 不强依赖 orchestrator，让 agent 自治判断下一步。
  • 把 README、progress 和日志设计成 LLM 可读的外部记忆。
  • 用高质量测试和结构化输出给 agent 提供方向盘。
  • 用 oracle 和差分测试把巨型任务切成可并行小任务。
  • 用角色化 prompt 让不同 agent 长期关注不同质量维度。
  • 最后保留人类验收，因为测试通过不等于工程完成。

如果说过去的 AI 编程是”让模型帮我写一段代码”，Agent Team 的新范式就是”为模型建一个能持续工作的工程环境”。真正的门槛不在模型会不会写代码，而在我们能不能把目标、反馈、边界和协作机制设计得足够清晰。
