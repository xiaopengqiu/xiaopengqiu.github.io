Claude Code /loop 机制深度解析：与 ralph-loop 的对比

背景：让 Agent 真正”跑起来”

Agent 最大的价值不在”一次问答”，而在持续运转——监控部署、轮询 CI、定期巡检、把一件复杂的事反复推进直到完成。Claude Code 在这条路上提供了两条官方路径：内置的 /loop 命令，以及社区插件 ralph-loop。两者目标相近（让同一 prompt 反复执行），机制却截然不同。

本文基于 Claude Code v2.x 二进制字符串、ScheduleWakeup / CronCreate 工具自描述，以及 ralph-loop 插件源码，把两者的状态管理、退出判断、效果验证逐一拆开，并指出一个常被忽略的事实：两者都不做真正的”任务完成度验证”——这是通用循环原语的刻意边界，不是疏漏。

一、/loop 的两种运行模式

/loop 的核心设计是两种模式二选一，由是否提供时间间隔决定：

形态 例子 底层机制 谁决定下一次触发时间
固定间隔模式 /loop 5m /babysit-prs CronCreate（5 字段 cron 表达式） 系统按 cron 表达式
动态自节拍模式 /loop check the deploy（无间隔） ScheduleWakeup（一次性唤醒） 模型自己选 delay
  • /loop 5m foo → 解析 5m 为 */5 * * * *，调用 CronCreate，prompt 字段存 foo。
  • /loop foo（无间隔）→ 立即把 foo 当一次正常 user turn 投喂，期望模型在回合结束前自己调用 ScheduleWakeup 安排下一次。

动态模式还有一个”自治”变体：用户根本没给 prompt（/loop 单独使用），此时用 sentinel <<autonomous-loop-dynamic>> 替代 prompt，运行时在 fire 时把它展开成完整的”autonomous-loop 指令”。CronCreate 模式有对应的 <<autonomous-loop>> sentinel，两者不可混用。

二、状态管理：不是”裸循环”

2.1 存储位置

  • 持久化（durable）：.claude/scheduled_tasks.json
    • 每条记录：{ id, cron, prompt, createdAt, recurring, createdBySessionId, createdByPid, createdByProcStart, lastFiredAt }
    • createdBySessionId / createdByPid / createdByProcStart 三元组用于区分是哪个会话创建的（多会话共存时的隔离）
    • lastFiredAt 在每次触发后被更新
  • 会话内存（session-only）：进程内 Map
    • CronCreate 默认 durable: false，ScheduleWakeup 总是 session-only
    • 退出 Claude 进程即丢失

2.2 动态模式额外的 per-prompt 状态

ScheduleWakeup 内部按 prompt 文本维护一个状态对象：

  • startedAt：这条 prompt 的循环何时开始（首次调用时设为 now）
  • lastScheduledFor：上次唤醒被定在何时
  • agedOut：是否已达到 7 天上限、被强制老化

每次 ScheduleWakeup(prompt, delay)：

  1. 先取消该 prompt 已有的 pending loop wakeup（kind === "loop" && prompt === $）—— 同一 prompt 任何时刻只有一个待唤醒。
  2. 计算 targetMs = now + delay*1000，向上取整到分钟边界（秒/毫秒清零）。
  3. 注册一个 kind: "loop" 的 cron 任务，cron 表达式就是 分 时 * * *（仅此一分钟匹配一次）。
  4. 更新 lastScheduledFor。

2.3 会话作用域

ScheduleWakeup、CronCreate(durable:false)、/loop 三者都属于 “session-scoped cron tasks”——这正是 Stop hook 输入里 session_crons 字段的含义。Stop hook 拿到这个列表，就能区分”会话真的结束了”和”会话在等下一次唤醒”。

三、退出循环的路径

3.1 固定间隔模式

  1. 用户主动取消：CronDelete <id>，或 /loop 命令的取消子命令。
  2. 7 天自动过期：recurringMaxAgeMs = 604800000（7 天）。过滤掉 createdAt + 7d < now 的任务。
  3. 会话结束：仅当 durable: false 时；进程退出即丢。
  4. 被 recurring 标志位为 false 的一次性任务：cron 字段编码单一触发时间，触发后自动移除。

3.2 动态自节拍模式

  1. 模型不再调用 ScheduleWakeup：这是最自然的退出——回合结束前不安排下一次，循环就停了。对应 telemetry tengu_loop_dynamic_wakeup_ends_turn。
  2. 7 天老化：if (now - startedAt >= recurringMaxAgeMs) { agedOut = true; return null; }。一旦 agedOut，后续 ScheduleWakeup 调用直接返回 null，不再注册新唤醒，并打 tengu_loop_dynamic_wakeup_aged_out 事件。
  3. 用户中断：X88() 取消所有 kind === "loop" 的 pending wakeup，并清理每个 prompt 的状态文件，打 loop_cancel_all。
  4. 会话结束：因为 ScheduleWakeup 是 session-only，进程退出即丢。

3.3 共同的保护机制

  • 动态模式的 delaySeconds 被 clamp 到 [60, 3600]（Math.max(60, Math.min(3600, ...))）。
  • 固定间隔 cron 最小粒度 1 分钟。
  • 这避免了”1 秒一次”把 API 打爆。

四、循环执行效果的”状态检验”——这一项两者都没有

这是关键，也是社区里最容易被误解的一点：

  • /loop 完全不做效果验证。它只负责在 REPL idle 时把 prompt 当新一次 user turn 灌进去，跑完就跑完，下一轮要不要继续完全交给模型自己看观察结果决定。调度器不读输出、不判断成败。
  • ralph-loop 也不做真验证。它判断”完成”的方式是字符串匹配 <promise>COMPLETE</promise>——这是模型声明“我做完了”，不是循环框架验证“你做完了”。模型说谎或自欺，框架不会发现。

也就是说：循环框架提供的是调度 + 终止信号通道，不是任务完成度评估器。真正的”效果是否达成”始终由模型自己感知和判断，框架不参与。把”完成判断”外包给模型是有意为之的轻量设计——通用原语刻意不持有任务域知识。

4.1 为什么”做不到”是设计选择

循环框架要判断完成，需要一个与任务无关的完成判据。但”完成”的语义每个任务都不同：bug 修复看测试、部署看 CI、文档看审阅、监控看告警消除。没有一个通用判据能覆盖所有任务，所以只有两条路：

  1. 硬编码一个判据（ralph-loop 的 <promise>COMPLETE</promise> 字符串）—— 等于让模型声明完成，框架不验证
  2. 完全交给模型（/loop 动态模式）—— 模型自己看环境决定续不续，框架只管调度

4.2 想要真验证，得在 loop 外面套一层

/loop 能当调度层，但 verifier 得自己接——比如 prompt 里要求模型每轮跑 npm test 并把结果写到约定文件，外部 watcher 读文件决定是否 CronDelete 取消循环。这就把”完成判断”从模型自觉提升到外部客观判据。

实际能落地的几种 verifier 形态：

验证形态 例子 loop 内置？
测试通过 SWE-agent 跑 unit test 否
状态查询 CI 状态 / API 返回 200 否
人工审批 human-in-the-loop gate 否
多 agent 互检 critic agent 复核 worker 输出 否
断言脚本 assert pytest.passed 否

五、动态模式深挖：模型即调度器

动态模式是 /loop 三种形态里最”智能”的——它把”何时续、何时停”的决策权交给了能读环境的模型。这里展开讲透。

5.1 何时进入动态模式

判断逻辑就一行：解析到时间 token → CronCreate；解析不到 → ScheduleWakeup 链。所以动态模式的触发条件就是”用户没给间隔，只给了一个要反复做的事”。

自治变体（/loop 单独使用）走 sentinel <<autonomous-loop-dynamic>>——它和带 prompt 的动态模式是同一机制，只是 prompt 字段填的是 sentinel，运行时在 fire 时把它展开成完整的”autonomous-loop 指令”。

5.2 如何使用

用户侧：

例如：

模型侧（这是动态模式的真正”使用方式”）——动态模式的核心契约是：每个回合结束前，模型必须自己决定要不要调 ScheduleWakeup，调了就续、不调就停。

三个要点：

  1. prompt 每轮原样回传——不是重新写一遍，是把 /loop 的输入 verbatim 传回去，下一轮 fire 时它就是新的 user turn。
  2. delaySeconds 被 clamp 到 [60, 3600]——低于 60 拉到 60，高于 3600 压到 3600。这是硬保护，防止”1 秒一次”打爆 API。
  3. 不调用 = 结束循环——这是动态模式最自然的退出路径。

5.3 实际跑起来是什么样

体验上：

  1. /loop check the deploy 一发出，立即当作一次正常 user turn 跑（不是等第一个间隔）。
  2. 模型跑完一轮（看 CI、查状态），在回合结束前调 ScheduleWakeup(prompt, 270, "CI 还在跑")。
  3. 270 秒后、REPL idle 时，调度器把同一个 prompt 灌回去 → 又一轮。
  4. 模型看 CI 绿了 → 不调 ScheduleWakeup → 循环自然终止。

5.4 选 delay 的启发式——动态模式”智能”的来源

模型按状态变化速度选间隔：

场景 delay 逻辑
状态快速变化（CI run、deploy） 60–270s 留在 5min cache TTL 内，命中缓存、便宜
状态慢变 / 真正空闲 / fallback 心跳 300–3600s 付 cache miss 代价换更长等待
无特定信号空转 1200–1800s（默认） 20–30 分钟 tick 一次
恰好 300s 避免 最差组合：付了 cache miss 又没摊销

关键提醒：harness-tracked 的活不要 poll。当 harness 能追踪的后台任务（Bash run_in_background、子 agent）完成时，系统会自动重新唤起你，此时再 ScheduleWakeup 短轮询是浪费。短间隔只用于”harness 看不到的外部状态”（CI run、deploy、remote queue），并配 fallback 长心跳兜底。

5.5 核心机制实现

底层：一次性 cron，不是 sleep。 ScheduleWakeup 不阻塞、不 sleep。它注册的是一个 kind: "loop" 的一次性 cron 任务，cron 表达式只有 分 时 * * *——仅匹配目标分钟那一次。触发后自动失效。

per-prompt 状态对象：每次 ScheduleWakeup(prompt, delay) 先取消该 prompt 已有的 pending loop wakeup（同一 prompt 任何时刻只有一个待唤醒），算 targetMs，向上取整到分钟，注册 cron，更新 lastScheduledFor。

cache 窗口对齐（动态模式独有的精细活）：

效果：用户选 270s delay，调度器会把实际触发时间往前拉到约 4:45 处，使其仍在 5 分钟 cache 窗口内 → 下一轮命中 prompt cache → 便宜。这正是工具描述里”60–270s 缓存命中”的来源。

7 天老化硬上限：

这是动态模式的安全网——模型忘记停也不会永远跑。

只在 REPL idle 时触发：调度器绝不在 mid-query 中途把 prompt 灌进去——必须等当前回合结束、REPL idle。配合 Stop hook 的协调：

当一个 loop cron 即将触发时，已挂的 Stop hook 的 block 决策被丢弃——避免 Stop hook 和 cron 同时灌 prompt 造成双重投喂。

5.6 固定间隔模式的抖动（避免雷鸣群）

为避免所有用户都在整点触发压垮 API，对触发时间加哈希抖动：

  • recurringFrac = 0.5、recurringCapMs = 1800000（30 分钟封顶）：recurring 任务的下次触发可往前最多拉 0.5×interval、最多 30 分钟。
  • oneShotMaxMs = 90000、oneShotMinuteMod = 30：一次性任务在 30 分钟整数倍时最多往前拉 90 秒。
  • 抖动量由 cron 任务 ID 前 8 位十六进制做哈希决定——每个任务的抖动是确定性的。

六、与 ralph-loop 插件的对比

官方 ralph-loop 插件实现的是同步、阻塞式循环：

  • 用 Stop hook 拦截 Claude 的退出尝试。
  • 每次模型想结束 → Stop hook 返回 { decision: "block", reason: <原始 prompt> } → 同一 prompt 被灌回 → 模型继续。
  • 状态文件 .claude/ralph-loop.local.md（YAML frontmatter：iteration、max_iterations、completion_promise、session_id）。
  • 退出条件：达到 max_iterations，或检测到 <promise>COMPLETE</promise> 字符串。
  • 完全在当前会话内同步发生，没有时间间隔概念——它不是”每 5 分钟查一次”，而是”不停干直到完成”。

/loop 则是异步、时间驱动的：

  • 不阻塞退出，REPL idle 时才触发。
  • 触发间隔由用户（固定）或模型（动态）决定。
  • 适合”周期性轮询状态”而非”持续干活到完成”。

两者可视为互补：ralph-loop 适合”把这件事做完”，/loop 适合”定期回头看一眼”。

6.1 三维度对照表

维度 /loop（固定间隔） /loop（动态模式） ralph-loop
驱动方式 时间（cron） 时间（一次性 cron） 事件（Stop hook 拦截）
同步/异步 异步 异步 同步阻塞
谁定节奏 用户写死 模型每轮选 不存在节奏（连续干）
状态存储 .claude/scheduled_tasks.json 或进程内存 进程内存 + per-prompt 状态 .claude/ralph-loop.local.md
自然退出 CronDelete 模型不调即停 <promise>COMPLETE</promise> 字符串
强制退出 7 天老化 7 天老化 max_iterations
cache 友好 否（整点触发常 miss） 是（模型选 60–270s） 否（连续会话）
效果验证 无 无（但模型可自感知） 无（字符串匹配伪验证）
适合场景 已知节奏的轮询 节奏取决于状态变化 把一件事干完为止

6.2 何时用哪个

  • /loop 5m ...：节奏明确（每 5 分钟查一次健康检查、每 10 分钟拉一次 PR 状态）。简单、可预测、可持久化跨会话存活。
  • /loop <task>（动态）：节奏取决于外部状态变化速度。CI 还要 3 分钟、部署还要 10 分钟——让模型自己挑 delay 最划算。这是离”循环 + 智能停”最近的形态。
  • ralph-loop：任务有明确的”完成”定义（修一个 bug、写一篇文章），希望模型持续干直到完成或达到上限。不关心时间间隔，只关心”做完”。

七、关键数据结构与常量速查

存储文件：.claude/scheduled_tasks.json

特性开关（feature flags，可被 config 覆盖）：

  • tengu_kairos_cron — 启用 CronCreate（默认 true）
  • tengu_kairos_cron_durable — 允许 durable 持久化（默认 true）
  • tengu_kairos_loop_dynamic — 启用动态 /loop（默认 false，灰度中）
  • tengu_kairos_loop_persistent — 动态模式的”persistent preamble”变体
  • CLAUDE_CODE_DISABLE_CRON 环境变量可整体关闭 cron
  • CLAUDE_CODE_LOOP_PERSISTENT 环境变量开启 persistent preamble

八、一句话总结

/loop = “把同一 prompt 按时间表反复投喂给模型” 的封装。

  • 给间隔 → 用 CronCreate 走 5 字段 cron，受 7 天上限保护。
  • 不给间隔 → 用 ScheduleWakeup，模型自己选 delay（60–3600 秒），不调用即终止。
  • 状态要么在 .claude/scheduled_tasks.json（持久）要么在进程内存（会话级）。
  • 每次触发只是把 prompt 当新 user turn 注入，效果检验由模型自己看观察结果决定。
  • 调度器自带抖动、cache 窗口对齐、Stop hook 协调，确保不压垮 API 也不与 Stop hook 冲突。

与 ralph-loop 对照：两者都有真正的状态管理和退出机制，但效果验证是两者的共同弱项——/loop 完全不做，ralph-loop 用字符串匹配伪验证。这不是”做得简单”，而是通用循环原语刻意把验证责任外包给上层。真要语义级完成判断，需要在循环外接一个 domain-specific 的 oracle（测试、状态探测、多 agent 互检、断言脚本），loop 本身只负责”按时把 prompt 灌回去”。

动态模式是 /loop 离”循环 + 智能停”最近的形态——它把”是否继续”的决策权留给了能读环境的模型，而不是死板地按 cron 触发。代价是完全依赖模型自觉：模型忘了调 ScheduleWakeup 就提前停了，模型一直调就跑满 7 天。这也是”效果验证靠模型自觉”的同一面硬币。
