Loop Engineering：从 Prompt 到 Loop，AI 工程的四次瓶颈迁移

本文总结自小林coding的微信公众号文章，梳理了从 Prompt Engineering 到 Loop Engineering 的演进脉络，拆解了一个 Loop 的五大组件，并分享了落地实操与三盆冷水。

原文链接：https://mp.weixin.qq.com/s/fhx_Lozs5G-sX11b7wnZgg
作者：小林coding
发布时间：2026年6月14日

背景：新词不断，到底在 engineering 什么？

AI 圈造新词的速度已经超过了学习速度：前年是 Prompt Engineering，去年是 Context Engineering，今年 3 月是 Harness Engineering，现在 6 月又来了 Loop Engineering。

但这不是营销轮换——每个新词的出现，背后都是同一件事：上一个瓶颈被解决了，新的瓶颈暴露出来了。

点火的三个关键人物在同一周说了同一件事：

  • Peter Steinberger（OpenClaw 作者）：「你不该再给 coding agent 打 prompt 了，你该去设计那个给 agent 打 prompt 的 loop。」
  • Addy Osmani（Google Cloud AI 总监）：在长文中给出了正式定义。
  • Boris Cherny（Claude Code 创始人）：「我已经不 prompt Claude 了，是 loop 在运行着 prompt Claude。我的工作是写 loop。」

Q1：四个 Engineering 的瓶颈迁移

模型每变强一截，瓶颈就往外移一层：

1. Prompt Engineering（2023）

模型只会一问一答，你问得好不好直接决定答得好不好。大家研究话术：角色扮演、思维链、少样本示例。瓶颈在「怎么说」。

2. Context Engineering（2025）

模型变强了，光会说话不够用。你让它改 bug，它没看过你的代码、不知道你的规范。瓶颈从「怎么问」移到了「喂什么」。

把对的代码、文档、工具、历史记忆，在对的时机塞进上下文窗口——这就是上下文工程。

3. Harness Engineering（2026 年初）

模型已经能连续干几个小时的活了，卡脖子的变成了工具、权限、沙箱、子 agent、上下文管理等运行环境。业内公式：Agent = 模型 + Harness。

4. Loop Engineering（2026 年中）

话术、材料、驾驶舱都解决了，最后的瓶颈是——你自己。模型在等你布置任务，harness 在等你启动。你睡觉，它就停工。Loop Engineering 瞄准的就是这最后一环：设计一个系统，让「下一次回车」不再由你来按。

Q2：Loop Engineering 到底是什么？

Osmani 的正式定义：

「Loop Engineering 就是把『亲自给 agent 写 prompt 的那个你』替换掉。你转而去设计那个代替你做这件事的系统。」

过去的协作方式是回合制——你写一条 prompt，读输出，再写下一条。Loop Engineering 说的是，把「发现任务、布置任务、检查结果、决定下一步」设计成一个能自己运转的循环。

打个比方：以前你是接线员，每个电话都要亲自接答；现在你是工单系统设计师——分流规则、办结标准、升级机制定好，系统自己转。

杠杆支点移动了：以前写一条好 prompt，收益是「这一次回答变好」；现在设计一个好 loop，收益是「之后每一次循环都变好」。投入从消耗品变成了资产。

Q3：一个 Loop 的五大件 + 记忆

1. 自动化——心跳

定时或事件触发。心跳有了，循环才算活着。

2. Worktree——让并行不变成打架

给每个 agent 一个独立的工作目录和分支，共享仓库历史但物理互不干扰。各干各的，最后各开各的 PR。

3. Skill——治好「金鱼记忆」

把项目知识写成文件放在仓库里，让 agent 该用的时候自己读。没有 skill，loop 每个周期都从零重新推导你的项目；有了 skill，知识是复利的。

4. Connector——摸到真实世界

基于 MCP 协议的连接器，把 issue 工单、监控、消息、PR 等外部系统接进来。一个完整的 loop 会自己开好 PR、关联工单、CI 变绿后通知人。

5. Sub-agent——写的人和查的人必须分开

写代码的那个模型，给自己的作业打分时，实在太手下留情了。

让 A 出方案，让干净上下文的 B 来挑刺。B 没有「希望自己是对的」的包袱，挑出来的问题才是真问题。

记忆——Loop 的命根子

模型在两次运行之间会忘掉一切。解法朴素：把记忆放在磁盘上（markdown 文件、任务看板），让它活在单次对话之外。

「Agent 会忘，但 repo 不会。」—— Osmani

Q4：一个真实 Loop 的样子

Osmani 给出的晨间 Triage Loop：

  1. 每天早晨自动触发，调用分诊 skill
  2. 读昨天的 CI 失败、未关闭 issue、最近提交，把值得处理的问题写进状态文件
  3. 对每个问题开隔离 worktree，派 sub-agent 起草修复
  4. 第二个 sub-agent 审查，对照 skill 规范和测试
  5. Connector 自动开 PR、更新工单
  6. 搞不定的问题丢进待办收件箱，等人来看
  7. 所有经过写回状态文件，明天从今天停下的地方继续

整个过程里，你只设计了一次，中间任何一步都没有写过 prompt。

Q5：现在就能搭——工具对照

部件 职责 Codex Claude Code
自动化 定时发现和分诊 Automations 面板 计划任务、/loop、hooks
Worktree 隔离并行任务 内置 worktree git worktree、隔离配置
Skill 固化项目知识 Agent Skills Agent Skills
Connector 连接外部工具 MCP Connectors MCP servers
Sub-agent 写查分离 配置文件定义子 agent 子 agent、agent teams
记忆 追踪进度 markdown / 工单系统 markdown / 工单系统

实操：30 秒搭出第一个 Loop

场景：PR 提了之后等 CI，挂了切回去改、推送、再等，一下午切了八次窗口。

拆开看：10m 是心跳，中间是任务，最后一句是停止条件加汇报。

两个使用边界：

  • /loop 活在当前会话里，关掉电脑就停了。要全天候运行需要用计划任务或云端 routines。
  • 最小 loop 只有心跳、任务和停止条件，worktree 和 sub-agent 可以后续按需加上。

部件是一件一件长出来的，不是一天配齐的。

Q6：三盆冷水

Osmani 本人直说：「现在还早，我是持怀疑态度的。」

冷水一：验证仍然归你

无人值守运行的 loop，也是无人值守犯错的 loop。检查 agent 嘴里的「done」只是一个声明，不是一个证明。

冷水二：理解债越顺滑涨得越快

Loop 交付你没写过的代码越快，「仓库里实际存在的东西」和「你脑子里真正理解的东西」之间的鸿沟就越大。这叫理解债（Comprehension Debt）——代码可能不烂，但你不知道它为什么是对的。

冷水三：认知投降最舒服也最危险

不再对产出有自己的观点，它给什么就收什么——这叫认知投降（Cognitive Surrender）。

「带着判断力去设计 loop，它是解药；为了逃避思考去设计 loop，它是助燃剂。同一个动作，相反的结果。」—— Osmani

总结

三句话浓缩全文：

  1. 四次瓶颈迁移：Prompt → Context → Harness → Loop，模型越强，瓶颈越往外移，最后移到了「亲自按回车的你」身上。
  2. 五大件 + 记忆：自动化是心跳，worktree 防打架，skill 治金鱼记忆，connector 摸到真实世界，sub-agent 写查分离，状态文件让明天接得上今天。
  3. 三笔账同时变大：验证、理解债、认知投降——工具分不出你是在加速还是在逃避，你自己分得出。

「两个人可以搭一模一样的 loop，得到完全相反的结果。一个用它在自己深刻理解的工作上加速，另一个用它彻底逃避理解工作。Loop 分不出区别，你分得出。」—— Osmani
