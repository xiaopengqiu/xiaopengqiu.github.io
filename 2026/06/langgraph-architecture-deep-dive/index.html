LangGraph 源码解析：Pregel 驱动的 Agent 编排架构与竞品深度对比

项目简介

LangGraph 是 LangChain 团队推出的基于**图论（Pregel 模型）**的有状态 Agent 编排框架，GitHub 11k+ 星，MIT 许可。它将 Agent 工作流建模为有向图，节点通过共享状态（Channel）进行通信，核心运行时保证了持久化执行和可恢复性。

与 CrewAI（角色/任务抽象）、AutoGen（对话协议）、Dify（低代码 DAG）等框架不同，LangGraph 选择了最底层的图原语抽象，将分布式系统的成熟理念（Pregel、BSP、Checkpoint）系统性地引入 Agent 编排领域。

二、核心设计详解

1. Pregel 执行引擎 — tick/after_tick 循环

源码位置：pregel/_loop.py

LangGraph 的执行引擎直接借鉴了 Google 的 Pregel 论文（BSP - Bulk Synchronous Parallel）。执行按**超步(superstep)**推进：

  • 每个超步中，所有就绪节点并行执行
  • 节点通过 Channel 写入更新
  • 超步结束时，Channel 聚合所有写入
  • 下一个超步根据新状态决定哪些节点激活

这与传统 DAG 框架（只支持拓扑排序执行）本质不同——它支持循环、条件分支和动态路由。

tick() 方法（核心超步）：

  1. 检查 step > stop limit → 返回 False（默认 recursion limit = 10007，而非文档中的25）
  2. 调用 prepare_next_tasks() 确定就绪任务
  3. 无任务 → status=”done”，返回 False
  4. 检查 interrupt_before → 可能抛出 GraphInterrupt
  5. 返回 True（继续执行）

after_tick() 方法：

  1. 收集所有任务写入
  2. apply_writes() 更新 Channel 并触发 checkpoint
  3. 发射 “values” 流式输出
  4. 捕获 DeltaChannel 的 exit-mode 写入
  5. _put_checkpoint({"source": "loop"}) 持久化
  6. 检查 interrupt_after → 可能抛出 GraphInterrupt

状态生命周期：input → pending → (tick循环) → done | interrupt_before | interrupt_after | out_of_steps | draining

3. Checkpoint 持久化（Durable Execution）

每个超步结束时，引擎将完整的 Channel 状态 + 待处理写入序列化为 Checkpoint。v4 格式结构如下：

这保证了：

  • 故障恢复：进程崩溃后从最近的 Checkpoint 恢复
  • Human-in-the-loop：interrupt() 暂停执行，持久化状态，等待人类输入后精确恢复
  • 时间旅行：可以回溯到任意历史 Checkpoint 并重新执行
  • 分支执行：从同一 Checkpoint fork 出多条执行路径

Durability 模式：

模式 行为 适用场景
sync 同步持久化，下一步开始前确保写入 强一致性要求
async 异步持久化，与下一步执行并行 性能优先
exit 仅在图退出时持久化 短任务，大幅减少I/O

DeltaChannel 增量序列化细节：

  • checkpoint() 始终返回 MISSING（不存储值）
  • 实际快照决策在 create_checkpoint() 中：当 updates >= snapshot_frequency（默认1000）或 supersteps >= 5000 时做全量快照
  • 恢复时调用 get_delta_channel_history() 沿父链回溯，收集历史写入并 replay

4. interrupt() 实现机制

源码位置：types.py:811-934

interrupt(value) 工作流：

  1. 从 config 获取 PregelScratchpad
  2. scratchpad.interrupt_counter() 递增索引
  3. 如果索引 < len(scratchpad.resume) → 返回已缓存的 resume 值
  4. 检查 scratchpad.get_null_resume(True)
  5. 无 resume 值 → raise GraphInterrupt(Interrupt(value, id))

重要细节：

  • GraphInterrupt 继承自 GraphBubbleUp（继承自 Exception），但被 _suppress_interrupt() 在根层捕获并抑制
  • 恢复时整个节点从头重新执行，通过 scratchpad 的 interrupt_counter 按调用顺序匹配 resume 值
  • PregelScratchpad 是一个 frozen dataclass，跟踪：step, stop, call_counter, interrupt_counter, resume列表, subgraph_counter

interrupt 的实现基于 Python 异常机制配合 Checkpoint，实现了可恢复的协程语义——这在其他框架中极为罕见。

5. Command 机制

map_command() 将 Command 转换为写入：

  • goto: Send对象 → (NULL_TASK_ID, TASKS, send)，字符串 → (NULL_TASK_ID, "branch:to:{node}", START)
  • resume: (NULL_TASK_ID, RESUME, resume_value)
  • update: (NULL_TASK_ID, k, v) 每个键值对

Command.PARENT 在根图层级会抛出 InvalidUpdateError，仅子图可用。

7. 子图组合机制

  • 通过 find_subgraph_pregel() 自动发现 Runnable 树中的 PregelProtocol 实例
  • Checkpoint 命名空间：NS_SEP="|" 分隔层级，NS_END=":" 分隔命名空间与任务ID
  • 父图通过 CONFIG_KEY_CHECKPOINT_MAP 传递 checkpoint_id 映射
  • 每个子图有独立的 Checkpoint namespace，状态隔离但可通过 Channel 通信

这使得复杂工作流可以模块化开发和测试。

8. 流式输出模式

模式 语义 典型用途
values 每步完整状态 全局状态监控
updates 增量更新 UI 局部刷新
messages LLM token 级流式 实时打字效果
custom 自定义事件 业务逻辑通知
debug 内部执行跟踪 调试与排错

SDK 层额外支持 "messages-tuple" 模式。通过 StreamTransformer 体系（ValuesTransformer, UpdatesTransformer, CustomTransformer, MessagesTransformer, LifecycleTransformer, SubgraphTransformer）处理协议事件。

LangGraph 的真正护城河

1. 每超步 Checkpoint + interrupt + 时间旅行三位一体

没有任何其他主流 Agent 框架同时具备这三个能力。CrewAI 和 AutoGen 没有内建持久化；Dify 和 Coze 有平台级存储但缺乏细粒度的 per-step checkpoint；Semantic Kernel 的 Process Framework 开始引入状态持久化，但远未达到 LangGraph 的完整性。

2. Channel/Reducer 状态模型——将状态管理从节点逻辑中完全解耦

这是 LangGraph 最被低估的设计。其他框架要么让状态在角色对话中隐式传递（CrewAI），要么依赖全局变量（Dify）。LangGraph 的 Channel 系统让开发者精确控制：信息流向、并发写入合并策略、字段生命周期。

3. Pregel/BSP 执行模型的独特性

截至2025年初，没有其他主流 Agent 框架采用 Pregel/BSP 模型。AutoGen 走了事件驱动路线，SK 走了 Planner 路线，其余都是 DAG 或对话循环。Pregel 模型天然支持并行、循环、动态路由——这些在 Agent 场景中极为常见的需求。

竞争对手已经追上的领域

  • 开发者体验：Dify/Coze 的可视化编辑器对非开发者更友好
  • 企业基础设施：SK 深度集成 Azure，Coze 背靠字节
  • RAG 原生支持：Haystack 的 Pipeline+Component 天然适合 RAG 管线
  • 多语言 SDK：SK 的 C# + Python 双语言支持

四、Durable Execution 竞争格局

在持久化执行领域，LangGraph 面临来自通用平台的竞争：

框架/平台 类型 Durable Execution 方式 与 LangGraph 关系
Temporal + AI 通用编排 Event Sourcing + Workflow Replay 可组合——有人用 Temporal 编排 LangGraph
Inngest 事件驱动平台 Event-driven durable functions 面向 Serverless，与 LangGraph 互补
Restate Durable Execution 平台 Journal-based replay + suspend/resume 最接近 LangGraph 的 interrupt 模式
DBOS 数据库驱动 Postgres-backed durable workflows 面向数据密集型，非 Agent 专用
Fixpoint Agent Durable Execution Checkpoint + replay 2024-2025 新兴，直接对标 LangGraph

Restate 是最值得关注的对手：

  • Journal-based execution：每个操作记录到 journal，崩溃后通过 journal replay 恢复
  • Suspend/resume：类似 LangGraph 的 interrupt，支持 ctx.run() + ctx.awakeable()
  • 确定性重放：journal 保证相同输入产生相同结果

但差异在于：Restate 是通用 Durable Execution 平台，不是 Agent 原生；缺少 Channel/Reducer 等高级状态抽象；不直接支持 Python LLM 生态。

五、总结

LangGraph 的核心竞争力在于将分布式系统的成熟理念（Pregel、BSP、Checkpoint）系统性地引入 Agent 编排领域，而非简单地串联 LLM 调用。

维度 LangGraph 优势程度 说明
Durable Execution 极强 无直接竞争者
循环/图结构 强 唯一原生支持 cycle + 状态的 Agent 框架
状态管理精度 强 Channel/Reducer 独一无二
开发体验 中等 高层 API 在追赶，仍比 CrewAI 陡峭
可视化/低代码 弱 Studio 不如 Dify/Coze
企业基础设施 中等 LangGraph Cloud 在成长，但不如 Azure 集成
RAG/知识库 弱 需要外部组件
社区/生态 强 LangChain 生态效应

LangGraph 的核心竞争力 = Pregel 图计算 + Channel 状态管理 + Checkpoint 持久化的三位一体。 这不是单一功能可以被复制，而是架构层面的组合创新。竞争对手可以追赶单个维度（循环、持久化、状态），但很难同时达到三者的深度整合。

最大风险：如果 Restate/Temporal 等通用 Durable Execution 平台进入 Agent 领域，加上一个更友好高层 API 的框架，LangGraph 的中间层定位可能受到挤压——上面被 Dify/Coze 的低代码吸引走业务用户，下面被 Temporal/Restate 的基础设施吸引走需要强持久化保证的工程团队。

最适合场景：需要长时间运行、有人类参与、状态复杂的 Agent 系统（如客服工单处理、多步骤研究助手、审批流程等）。
