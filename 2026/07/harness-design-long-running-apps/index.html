Harness 工程分析——Anthropic 长时自主应用架构解读

文章：Harness Design for Long-Running Apps（作者：Prithvi Rajasekaran，Anthropic Labs 团队）

一、文章要解决的问题

让 Claude 在无人干预下完成两件事：

  1. 产出高质量前端设计（主观品味驱动）
  2. 构建完整全栈应用（可验证的正确性驱动）

这两个领域看似相反（一个靠主观判断、一个靠客观验证），但作者发现它们共享同一类失败模式，因此可以用同一套 harness 思路突破。

二、Harness 工程的核心思想

1. 从 GAN 借鉴的”生成-评估”对抗结构

核心创新：把单 Agent 的”自评自审”拆成两个独立 Agent —— Generator（生成器）与 Evaluator（评估器）。

  • 单 Agent 自评时倾向于”自信地夸自己”，即便输出明显平庸也打高分；
  • 把评估独立出来后，调教一个”挑剔的外部评估器”远比让生成器自我批判容易；
  • 一旦有了具体的外部反馈，生成器就有了可迭代的目标。

2. 三智能体架构（Planner + Generator + Evaluator）

角色 职责 关键设计
Planner 把 1–4 句话的 prompt 扩展为完整产品 spec 只约束”交付物”，不规定实现细节，避免错误级联；主动嵌入 AI 能力
Generator 按 spec 一次实现一个 feature sprint 制；每个 sprint 结束自评后交给 QA
Evaluator 用 Playwright MCP 像真人一样点击运行中的应用 按 sprint contract 逐条打分，低于阈值即打回

3. 两条贯穿始终的工程教训（来自早期 harness 工作）

  • 任务分解：把大构建拆成可控的小块；
  • 结构化 artifact 传递上下文：用文件在 session 之间交接状态（而非靠对话记忆）。

三、两个核心失败模式及对应解法

失败模式 1：上下文膨胀 / 上下文焦虑（Context Anxiety）

  • 现象：context 填满后模型失忆；或模型”自以为快到上限”而提前收尾。
  • 解法：Context Reset —— 清空 context、启动新 agent，通过结构化 artifact 把前一个 agent 的状态和下一步交接过去。
  • vs. Compaction：compaction 是原地摘要、保持同一 agent 继续；reset 给的是”清白起点”。Sonnet 4.5 的 context anxiety 严重到 compaction 都不够，必须 reset。代价：编排复杂度、token、延迟上升。
  • 演进：Opus 4.5 起该行为大幅消失，reset 在新模型上被整个移除，改为单 session + SDK 自动 compaction。

失败模式 2：自我评估失真

  • 现象：Agent 评价自己作品时系统性偏正面，主观任务（设计）尤甚。
  • 解法：Generator / Evaluator 分离。分离本身不消除宽容倾向（评估器仍是 LLM），但把”挑剔”做成可调教的目标，比让生成器自我批判现实得多。

四、前端设计：把主观品味变成可打分项

作者设计了 4 条评分准则，同时给到 Generator 和 Evaluator：

  1. Design Quality —— 整体是否像”一个作品”而非零件堆砌（色彩/排版/布局/意象共同形成 mood）；
  2. Originality —— 是否有自定义决策，还是模板/库默认/AI 套路（如”紫渐变+白卡片”被点名扣分）；
  3. Craft —— 技术执行力（层级、间距、对比度、色和谐）；
  4. Functionality —— 可用性，与美观无关。

关键加权：Claude 默认 Craft / Functionality 已经不错，但 Design / Originality 偏平淡 → 准则显式惩罚”AI slop”通用模式，加权 Design & Originality 推动”审美冒险”。

校准：用 few-shot 详细分数拆解校准评估器，减少跨轮次漂移。

循环：Generator 产出 HTML/CSS/JS → Evaluator 用 Playwright MCP 实际打开页面、截图、研究后打分写批评 → Generator 决策”继续精炼”或”彻底换方向”。5–15 轮/次，单次完整跑可达 4 小时。

意外发现：准则措辞本身就会塑形输出 —— 比如”the best designs are museum quality”这类话让生成器向某种视觉风格收敛，说明prompt 用词直接塑造了输出性格。

亮点案例：荷兰美术馆网站，第 9 轮还是一个干净的暗色 landing page，第 10 轮推翻重来，变成 CSS 3D 透视房间 + 墙上自由挂画 + 门洞式导航 —— 单次生成里罕见的创造性跳跃。

五、全栈编码：三 Agent 完整流程

关键机制：Sprint Contract（sprint 契约）

在每个 sprint 之前，Generator 与 Evaluator 协商一份契约：约定这块工作”done”长什么样、怎么验证。

  • 存在原因：spec 故意写得高层，需要一个桥梁把 user story 落到可测实现；
  • Generator 提案（建什么、怎么验证）→ Evaluator 审核（方向对不对）→ 迭代到达成共识 → 才写代码；
  • 通信方式：全部通过文件（一个写、另一个读并响应），保证上下文可追溯、不依赖对话历史。

实测对比（同一 prompt：2D retro game maker）

Harness 时长 成本
Solo（单 Agent） 20 min $9
Full harness（三 Agent） 6 hr $200

成本 20×，但质量差异立竿见影：

  • Solo：界面能跑、流程僵硬、entity 出现在屏幕上但完全无响应——核心玩法坏了，且代码里看不出断在哪；
  • Harness：Planner 把一句话扩展成 16 个 feature / 10 个 sprint 的 spec（含精灵动画、行为模板、音效、AI 辅助精灵生成与关卡设计、导出分享）；Play mode 真的能玩（虽有小瑕疵如角色穿模）。

Evaluator 的实际产出（举例）

评估器不是泛泛而谈，而是定位到文件+行号：

  • fillRectangle 函数存在但未在 mouseUp 触发；
  • LevelEditor.tsx:892 删除键条件错配，应为 selection || (selectedEntityId && activeLayer === 'entity')；
  • PUT /frames/reorder 路由定义在 /{frame_id} 之后，FastAPI 把 reorder 当 frame_id 解析返回 422。

Sprint 3 一个 sprint 就有 27 条验收准则。

调教评估器的反模式

开箱即用的 Claude 是个差的 QA：会先识别出问题，然后**自我说服”不是大事”**放行；测试偏浅，漏掉边界。调教循环：读日志 → 找判断分歧样本 → 改 QA prompt → 反复迭代，直到评估器的判断与人一致。

六、Harness 的演进与简化（关键工程洞察）

“harness 里每个组件都编码了一个’模型自身做不到’的假设，这些假设值得反复压力测试——既因为它们可能是错的，也因为模型一升级就可能过时。”

简化原则

“find the simplest solution possible, and only increase complexity when needed”（出自 Building Effective Agents）。

第一次激进简化失败

砍到只剩骨架 + 加新想法 → 性能掉得无法复现，且分不清哪些组件是 load-bearing。

→ 改为一次只拆一个组件、观察影响的方法论。

Opus 4.6 带来的简化机会

4.6 官方改进：planning 更稳、长任务持续更久、大代码库更可靠、自查/调试更强、长上下文检索更好 —— 这些正是 harness 之前补的短板。

具体动作

  1. 移除 sprint construct —— 4.6 原生能管理大任务，不再需要外部分解；
  2. 保留 Planner（去掉后 Generator 会欠 scope、直接开建、feature 变少）；
  3. 保留 Evaluator，但从 per-sprint 改为单次 end-of-run —— 因为模型变强后，”评估器是否必要”取决于任务是否落在模型能独立搞定的边界之外；
  4. 新增：让 Generator 真正构建一个能通过工具驱动应用自身功能的 agent（这部分知识训练数据覆盖薄，需要大量迭代调教）。

DAW（数字音频工作站）实测

prompt：Build a fully featured DAW in the browser using the Web Audio API.

阶段 时长 成本
Planner 4.7 min $0.46
Build R1 2h 7min $71.08
QA R1 8.8 min $3.24
Build R2 1h 2min $36.89
QA R2 6.8 min $3.09
Build R3 10.9 min $5.88
QA R3 9.6 min $4.06
合计 3h 50min $124.70

Generator 连续跑了 2 小时+ 无需 sprint 分解仍保持连贯。QA 仍抓到实质缺口（clips 不能拖动、录音只是按钮空转、效果器是数字滑块而非图形 EQ 曲线），证明评估器在”最后一公里”仍有价值。

局限：Claude 听不见声音 → 音乐品味的反馈环失效。

七、技术难点总结

  1. Context Anxiety 的识别与权衡：reset 解决了焦虑但带来编排/token/延迟成本；判断”何时 reset、何时 compaction”需要对模型行为的实证观察（Sonnet 4.5 必须 reset，Opus 4.5/4.6 不需要）。

  2. 主观品质的可量化：把”美不美”转成”是否符合设计原则”，再用 few-shot 锚定评估器偏好，避免分数漂移。准则措辞本身会塑形输出，是个杠杆也是个雷。

  3. Evaluator 的调教难度：开箱 Claude 是差的 QA（自我说服放行、测试偏浅），需要”读日志 → 找分歧 → 改 prompt”的多轮开发循环，且要让它定位到文件+行号才有可执行性。

  4. Load-bearing 组件的识别：砍组件要一次一个、对照观察，否则性能掉下来分不清是哪个组件的功劳。这是 harness 维护的元方法论。

  5. Spec 粒度的平衡：Planner 故意只写高层 spec（怕错级联），但又需要 sprint contract 把高层 spec 桥接到可测实现 —— 两层抽象分工明确。

  6. 跨 session 上下文交接：用文件通信而非对话，保证 artifact 可追溯、可被新 agent 独立读取。

  7. 模型升级 vs. Harness 复杂度的反向关系：模型越强，某些 harness 组件从”必需”变”冗余”，但新模型也打开了”做更难任务”的空间，需要新增组件去填补新边界 —— harness 的”有趣组合空间”不会随模型进步而收缩，只会迁移。

八、一句话核心结论

用 GAN 式的”生成-评估”分离 + 结构化 artifact 跨 session 交接，把长时自主编码从”单 Agent 自评自嗨”变成”Planner 规划、Generator 实现、Evaluator 挑刺”的可迭代闭环；模型变强时主动拆除已过时的脚手架、保留仍有价值的部分，并在新能力打开的边界上重新加件 —— 这是 harness 工程的元方法论。

值得收藏的工程金句：

“harness 中每个组件都是关于’模型做不到什么’的假设，模型升级时这些假设最值得被压力测试。”
